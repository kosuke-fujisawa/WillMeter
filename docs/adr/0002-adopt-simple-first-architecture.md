# 0002. Simple First を既定の設計方針として採用する

- 日付: 2026-07-12
- 状態: 採用

## 背景

WillMeter の `CLAUDE.md` は、Clean Architecture と DDD に基づく4層構造、Repository、UseCase、Domain Events などの一律適用を求めていた。一方、共通の開発方針は「Simple First」とし、レイヤー分割や抽象化は具体的な必要性が生じてから導入することを求めているため、両者が衝突していた。

WillMeter は現在、ウィルパワーの表示・消費・回復・リセット、永続化、多言語切り替えを中心とする小規模な iOS アプリである。既存コードには Domain / Application / Infrastructure / Presentation のディレクトリと Repository protocol があるが、リポジトリ固有の例外として今後も全DDDパターンを強制するだけの複雑性は確認できない。また、従来の文書には未実装機能や検証不能な品質評価が混在し、実装との乖離が生じていた。

## 選択肢

1. WillMeter を共通方針の例外とし、Clean Architecture + DDD の4層構造を維持・強制する
2. 共通の Simple First に統一し、DDDの語彙やパターンは効果が明確な箇所だけ選択的に使う
3. 既存コードを直ちにフラットな構造へ全面移行する

## 採用案

選択肢2を採用する。

ロジックと UI・永続化などの I/O の分離、ユビキタス言語、効果の明確な Value Object は維持する。一方で、4層構造、Repository、UseCase、Domain Events、境界づけられたコンテキスト、集約などを形式的には強制しない。抽象化は、複数実装、テストでの差し替え、外部境界の隔離など、具体的な必要性が生じた時点で導入する。

既存のディレクトリ構造と抽象化は、この決定だけを理由に一括削除しない。機能変更やリファクタリングの範囲で、テストを保ちながら段階的に単純化する。

## 採用理由

- 現在のプロダクト規模では、全DDDパターンの強制による認知負荷が便益を上回る
- 共通方針との例外をなくし、作業者ごとの設計判断のぶれを減らせる
- ロジックと I/O の分離を残すことで、テスト容易性という既存設計の主要な利点は維持できる
- 全面移行を避けることで、動作中のコードを設計方針変更だけで大きく書き換えるリスクを抑えられる
- 必要になった抽象化を後から追加できるため、この判断は変更しやすい

## 欠点・リスク

- 既存の4層構造と新しい軽量方針が当面併存し、どこまで単純化するか個別判断が必要になる
- レイヤー規則による機械的な配置判断がなくなるため、責務分離をコードレビューで確認する必要がある
- 将来ドメインが複雑化した場合、境界や集約を改めて設計するコストが発生する
- 「Simple」を理由に UI へロジックを戻す誤用が起こり得るため、ロジックと I/O の分離は明示的に守る必要がある

## 再評価条件

次のいずれかが生じた場合、この決定を見直す。

- 複数の独立したドメイン領域が生まれ、用語や整合性ルールが衝突する
- 永続化・同期・外部APIなどの実装が複数になり、明確な境界が必要になる
- 複数チームが並行開発し、モジュール間の契約を固定する必要が生じる
- 軽量構造が原因で、同種の責務混在や回帰が繰り返し発生する

## 影響範囲

- `CLAUDE.md` の設計・実装ガイドライン
- 今後追加・変更する Swift コードとテスト
- 既存の Domain / Application / Infrastructure / Presentation 構造（即時変更なし）
- README や設計資料に残る「Clean Architecture + DDD を厳格適用」という説明（別途、実装状況に合わせて更新対象とする）
