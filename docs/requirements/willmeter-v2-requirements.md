## 1. プロダクト概要(PRD相当・圧縮版)

### 1.1 コンセプト

WillMeterは、人が一日の中でどのようにウィルパワー(意志力)を消費・回復しているかを記録・可視化し、「本当にやりたいことへ意志力を使える生活」を支援するアプリである。ToDoアプリ・時間管理アプリ・集中タイマーではなく、**意志力という有限資源を管理する「意思力家計簿」**である。

### 1.2 Mission / Vision / Value

- **Mission**: 人が本当にやりたいことへ意志力を使える社会をつくる。
- **Vision**: 人は「時間」ではなく「意志力」によって人生が決まる、という認識を広める。
- **Value**: 可視化(見えない意志力を見えるようにする)/ 気付き(何に意志力を使っているか知る)/ 節約(不要な消費を減らす)/ 最適化(本当にやりたいことへ配分する)。

### 1.3 解決したい課題

現代人は時間ではなく意志力が不足している。SNS・ショート動画・通勤・会議・判断・通知などが少しずつウィルパワーを消費し、夜には本当にやりたかった勉強・読書・創作・運動に使う意志力が残っていない。WillMeterはこの消耗過程を可視化する。

### 1.4 ターゲットユーザー

- **メイン**: エンジニア、デザイナー、クリエイター、学生、自己啓発に関心がある人、ADHD傾向がある人、集中力を改善したい人、Forest利用者。
- **サブ**: 経営者、管理職、研究者、ライター、フリーランス。
- **想定しない**: 意志力不足を感じていない人、退屈で刺激の少ない生活を送る人、自己分析に関心がない人。

注意: マーケティング文言・ストア説明でも「ADHD」等の診断名を対象者として名指しすることは避ける(4.3節参照。診断名による対象化を避ける方針)。

### 1.5 設計思想

- 静かなUI、広告なし、通知なし、ユーザー主導、シンプル操作、自己観察、強制しない。
- **通知しない理由**: 通知もウィルパワーを消費するため。必要なときだけ開くアプリとする。
- **広告を出さない理由**: 広告は意図せず注意を奪い、ウィルパワーを消費するため。広告モデルは採用しない。
- **Forestとの関係**: WillMeterは集中時間を測るアプリではなく、意志力そのものを管理する。Forestは誘惑を遠ざける。両者は競合ではなく補完関係。

### 1.6 料金体系

- **無料**: 現在値表示、消費・回復記録、日次履歴、カテゴリ/サブカテゴリ編集、初期設定・再設定、モノクロモード、Tips、クレジット。
- **プレミアム(買い切り、参考価格480円。正式価格はApp Store上の価格設定を正とする)**: 週次分析、月次分析、カテゴリ分析、時間帯分析、頻度分析、追加テーマカラー、データエクスポート。
- 広告・サブスクリプションは採用しない。

### 1.7 非機能要件(概要)

- オフライン動作(ネットワークが必要なのはプレミアム商品情報取得・購入・購入復元のみ)。
- 通常起動は2秒以内を目標(1秒以内は必須要件としない)。
- 記録確定後、ユーザー操作を待たずに保存する。保存失敗時はメモリ上の状態を保持しつつ再試行し、失敗が継続する場合は「データが保存されていない」ことを画面上に明示する(「クラッシュさせない」だけでは不十分、データ消失防止を優先)。
- アクセシビリティ: Dynamic Type、VoiceOver、十分なタップ領域、色以外による状態表現、モノクロモード、Reduce Motionへの配慮、コントラスト確保。
- プライバシー: アカウント登録・サーバー送信・行動分析SDK・広告SDK・クラッシュ解析SDK(既存ADR 0005によりApple標準クラッシュレポートを使用、詳細は6章)・外部AI送信・位置情報/連絡先/ヘルスデータ取得のいずれも行わない。

### 1.8 ライセンス方針

ソフトウェアライセンスとアプリ内コンテンツ(文章・Tips・画像)のライセンスを分離して検討する。

- ソースコードのライセンス候補: MIT / Apache 2.0 / proprietary(**未決定**)。
- アプリ内コンテンツのライセンス候補: CC BY-NC 4.0(**未決定**)。
- 注意: 現在リポジトリの `LICENSE` は CC BY-NC 4.0 (ソフトウェア向けとしては非典型)。本仕様確定時に正式決定し、`LICENSE` / README / クレジット画面の記載を揃える(7章の未確定事項にも記載)。

### 1.9 将来構想(初期リリース対象外)

Android版、Web版、Apple Watch対応、Widget対応、iCloud同期、複数端末同期、アカウント登録、SNS共有、AI分析、コミュニティ機能、自由記述メモ、位置情報記録、ローカル/Push通知、他者との比較・ランキング・連続記録報酬、自動行動検出、ヘルスケア/スクリーンタイム連携。

## 2. ドメインモデル(確定仕様)

### 2.1 ウィルパワーの定義

> 意思決定、注意の維持、行動の開始、誘惑への抵抗、感情の調整などに利用される、本人が主観的に認識する心理的な余力。

医学的診断・厳密な心理測定を目的としたものではなく、利用者の自己観察を数値化したものである。WillMeterは行動を善悪で判定しない(同じ行動でも人によって消費にも回復にもなり得る)。

### 2.2 数値モデル

| 項目 | 仕様 |
|---|---|
| 現在値の型 | Int |
| 現在値の内部範囲 | 実用上の制限なし(保存時の防御範囲として `-999〜999` を推奨) |
| 1回の変化量(サブカテゴリ設定値) | `-10〜-1`(消費)または `+1〜+10`(回復)。`0`は設定不可 |
| メーター標準表示範囲 | `-10〜+10`(目安であり最大値・最小値ではない) |
| 表示範囲超過時 | 実数値をそのまま表示し、メーター図形は上限/下限位置で止め「+10超」「-10未満」等を補助表示する。表示範囲は自動拡張しない(同じ値でも日によって見た目の大きさが変わり比較しにくくなるため) |

**「最大ウィルパワー」という概念は採用しない。** 代わりに利用者が任意設定する「**好調時の目安**」(入力範囲 `1〜10`)を用いる。好調時の目安はメーター上の補助線・自己比較・将来の傾向分析に使うが、**現在値の上限としては使わない**。

### 2.3 初期設定(オンボーディング)

「初期診断」という名称は使わない(医学的診断との誤解を避けるため)。UI上の名称は「初期設定」「現在値を設定」「基準を設定し直す」等とする。

画面順序:

1. アプリの説明(客観測定ではなく自己観察の目安であることを明記)
2. 「現在のウィルパワーはどのくらいですか?」(入力範囲 `-10〜+10`。初回から負の値を許容する)
3. 「十分に余力があると感じる状態はどのくらいですか?」(入力範囲 `1〜10`、好調時の目安)
4. 設定完了(後から変更できる旨を明示)

再設定は設定画面から実行でき、「基準値のみ変更」「現在値と基準値の両方を変更」を選択可能。再設定・再診断は過去の行動記録を変更しない。

### 2.4 行動記録モデル

行動記録(ActivityLog)は1回の消費または回復を表す。少なくとも次を保持する: 記録ID、記録日時、所属する「WillMeter上の日」、消費/回復の別、カテゴリ、サブカテゴリ(記録時点の名称・変化量のスナップショットを含む)、変化量、記録前の現在値、記録後の現在値、作成時刻、更新時刻、並び順(`recordedAt` + `sequenceNumber`)、取消状態。

基本フロー: `消費/回復を選択 → カテゴリ選択 → サブカテゴリ選択 → 記録確定`(サブカテゴリ選択と同時に確定。標準3タップ、確認画面は挟まない)。

- 消費入力では負のサブカテゴリのみ、回復入力では正のサブカテゴリのみを表示する。
- 同一カテゴリ内に消費・回復のサブカテゴリが共存してもよい(カテゴリ自体は方向を持たない。2.6節参照)。
- 入力直後、画面下部に一時的なフィードバック(例:「電車通勤 -3 を記録しました [取り消す]」)を表示する。通知としては出さない。
- 直前入力の取消を提供する(初期リリース対象、2.8節参照)。

### 2.5 手動補正

現在値と体感が大きくずれた場合に現在値を手動補正できる機能を、初期リリースに含める。

- UI名称: 「現在値を調整」(メーター画面の補助メニューから)。
- 入力方式: **調整後の現在値を指定する方式に統一**(差分指定はしない)。
- 記録種別 `adjustment` として通常の行動記録と区別する。補正前値・補正後値・差分・補正日時を保持する。
- **カテゴリ分析には含めない**。値の推移グラフには含め、補正であることが分かる記号を付ける。

### 2.6 カテゴリ / サブカテゴリモデル

- **カテゴリ自体には方向(消費/回復)を持たせない**。方向と変化量は**サブカテゴリ**が持つ。これにより同一カテゴリ内に消費・回復のサブカテゴリを共存できる。
- カテゴリ・サブカテゴリは利用者が追加・名前変更・削除・並び替えできる。
- **削除は論理削除**(`isActive = false`)とする。過去に使用された名称・値は行動記録側にスナップショットとして保存し(`categoryNameSnapshot` 等)、削除後も過去履歴の表示は壊れない。
- **サブカテゴリの変化量を変更しても過去の行動記録には遡及しない**。行動記録は記録時点の変化量を保持する。
- システム種別(利用者が名称を変更しても内部種別は維持): `normal`(通常) / `nap`(仮眠) / `dayEndingSleep`(一日を終える睡眠) / `adjustment`(手動補正) / `dayStart`(一日の開始記録)。
- 初期プリセットカテゴリ(10件、選択負荷を考慮した推奨セット): 睡眠、休憩、食事、運動、仕事、通勤、家事、動画・SNS、対人交流、創作・学習。カテゴリ名・サブカテゴリ名・初期値は公開前にユーザーテストで調整する前提の仮値であり、断定的な絶対値ではない(2.1節の思想に基づく)。

### 2.7 睡眠と一日の境界

一日は**午前0時では自動終了しない**。一日の終了は、利用者が「一日を終える睡眠」を記録した時点で行う。徹夜時にも特別なリセット処理はしない(既存ADR 0001と整合。6章参照)。

睡眠は2種類に統一する(「寝落ち」を独立種別にはしない。仮眠または一日終了のどちらかとして扱う):

- **仮眠**: 通常の回復記録として扱う(回復値を持つ)。一日(WillMeterDay)は終了しない。
- **一日を終える睡眠**: 日境界イベントである。**回復値を直接は持たない**(睡眠自体に固定の回復量を加算しない)。現在のWillMeterDayを完了状態にする。

一日を終える睡眠を記録した直後には翌日の開始値を入力させない(睡眠前に翌朝の状態は分からず、その場での入力は推測値になるため)。睡眠記録後、アプリは `awaitingDayStart` 状態になり通常の消費・回復入力はできなくなる(誤操作時のために「睡眠記録を取り消す」導線は用意する)。**次回起動時**に「目覚めた現在のウィルパワーはどのくらいですか?」と問い、その回答を新しいWillMeterDayの開始値とする。

一日終了処理の手順: (1) 睡眠の回復記録を保存 → (2) 現在のWillMeterDayを完了状態にする → (3) 終了時刻を保存 → (4) その日の集計値を確定 → (5) 翌日の開始値確認へ進む → (6) 新しいWillMeterDayを開始する。

### 2.8 過去記録の編集・初期リリース範囲

初期リリースで可能にする範囲:

- 直前入力の取消
- 当日記録の削除
- 当日記録の時刻変更

初期リリース対象外(将来対応):

- 過去日の記録編集
- 過去の日境界修正
- 過去日の開始値変更

理由: 後続値の再計算と日境界の再構築が複雑であり、初期リリースの範囲を超えるため。ただしデータモデル上は将来の再計算に対応できる構造にしておく(記録前値・記録後値を唯一の真実とせず、差分から再計算できるようにする)。

### 2.9 WillMeterDay(一日)モデル

属性: `id`, `startedAt`, `endedAt`, `startValue`, `endValue`, `status`(`active` / `completed`), `createdAt`, `updatedAt`。

- 開始値は「初回設定」「睡眠後の自己申告」「手動での日開始」「日境界修正」のいずれかで設定される。
- 終了値はその日最後の行動記録後の現在値。翌日開始値(自己申告)とは別に保持する。
- **活動中(`active`)のWillMeterDayは常に最大1件**。同時に複数の活動中の日を作らない。
- 1つのWillMeterDayが複数のカレンダー日付をまたぐことを許容する(例: 開始 7/20 08:00〜終了 7/21 01:30 は WillMeter上は1日として扱う)。

### 2.10 履歴・分析モデル

**無料機能: 日次履歴**。表示内容は開始/終了時刻、開始値/終了値、総消費量、総回復量、行動記録一覧、カテゴリ別集計。

- 総消費量 = その日の負の変化量の絶対値合計。総回復量 = 正の変化量の合計。純増減 = 総回復量 − 総消費量。手動補正は純増減に含めるかどうかを表示上区別する。
- 日次グラフは円グラフを使わない(時間的推移を表しにくいため)。**値の推移は折れ線グラフ、消費・回復の内訳は横棒/積み上げ棒**で表現する。

**プレミアム機能**:

- **週次分析**: 直近7日ではなく**暦週**を基本とする。週の開始曜日は端末ロケールに従い、日本語環境では月曜開始を初期値とする。
- **月次分析**: 暦月単位。
- **カテゴリ分析**: カテゴリ単位で記録回数、総消費量、総回復量、平均変化量、最大変化量、利用日数、時間帯別分布を集計。同一カテゴリの消費・回復は分けて表示。
- **時間帯分析**: 記録の実時刻をもとに時間帯ごとの傾向を集計。初期区分は 朝(05:00-10:59) / 昼(11:00-16:59) / 夜(17:00-22:59) / 深夜(23:00-04:59)。将来変更可能。
- **頻度分析**: 記録頻度の高い順・消費/回復行動の頻度・期間比較。頻度は行動の良し悪しを示さない。

分析結果には断定的な助言(「この行動は悪い」「不健康」「意志力が低い」等)を使わず、「多く記録されています」「この時間帯に消費が集中しています」のような傾向表現に限定する。手動補正・日開始記録・システム移行データ・削除/取消済み記録はカテゴリ分析から除外する(日次の値推移には含めてよい)。記録が少ない期間も分析画面自体は表示し、「この期間には、分析できる記録がまだありません」等の非強制的な表現にとどめる。

### 2.11 プレミアム権利モデル

- 買い切り型の非消耗型アプリ内課金1点のみ(StoreKit 2)。参考価格480円、正式価格・商品IDはApp Store Connect登録後に確定。
- 対象: 週次/月次/カテゴリ/頻度/時間帯分析、テーマカラー、データエクスポート。
- 購入情報が一時的に確認できない場合でも、ローカルに有効な購入履歴がある利用者の機能を即座に無効化しない。購入復元を提供する。
- 全データ削除操作の対象に購入権利は含めない(StoreKit上の購入情報から復元する)。

### 2.12 テーマ・アクセシビリティ

- 無料: 淡い青の既定テーマ(消費=赤系、回復=緑系、中立=青系)。**モノクロモード**も無料提供。モノクロモードでは色だけに依存せず `+`/`-` 記号・ラベル・アイコン・線種・パターン・数値を併用する。
- プレミアム: 複数カラーテーマ(表示のみへ影響し、履歴や数値には影響しない)。

### 2.13 Tips・クレジット

- Tipsはクレジット画面の「意志力を減らさないヒントを見る」ボタンからのみ開く(自動表示・通知はしない)。属性: ID、タイトル、本文、表示順、有効/無効、バージョン、カテゴリ。初期リリースではローカルの静的データ(JSONまたはSwiftリソース)で保持する。
- Tips本文は断定ではなく提案として記述する(例: 「〜と感じる人もいます」であり「〜すると必ず減ります」は避ける)。
- クレジット画面には WillMeterについて / プレミアム導線(「気に入っていただけたら支援としてプレミアムをご検討ください」、ただし「寄付」という表現は使わない。課金は寄付ではなく機能購入) / ライセンス / Tips導線 を含める。

### 2.14 データエクスポート

プレミアム利用者は自己データを外部出力できる。**初期版はCSVを優先**(表計算ソフトで確認しやすく、実装範囲を限定できるため)。完全バックアップ用JSONと再インポート機能は将来対応。出力対象は WillMeterDay、ActivityLog、Category、SubCategory、設定、基準値、エクスポート日時、データ形式バージョン。エクスポートは利用者の明示操作でのみ行い、自動アップロードはしない。

### 2.15 リセット

- **現在値のみのリセット**: 過去履歴は保持したまま、初期設定または現在値の自己申告により再設定する。
- **設定のリセット**: カテゴリ・テーマを初期状態へ戻す。履歴を残すか消すかは利用者へ明示する。
- **全データ削除**: 行動履歴、WillMeterDay、カテゴリ/サブカテゴリ設定、現在値、基準値、表示設定を削除する。購入権利は対象外。確認操作を必須とする。

### 2.16 データ不変条件(常に満たすべき条件)

1. 活動中のWillMeterDayは最大1件である
2. 行動記録は必ず1つのWillMeterDayへ所属する
3. 通常記録の変化量は0ではない
4. 消費入力の変化量は負である
5. 回復入力の変化量は正である
6. 記録後の値は、記録前の値と変化量から算出できる
7. カテゴリ削除後も過去履歴を表示できる
8. 設定値変更は過去記録へ遡及しない
9. 再設定は過去履歴を変更しない
10. 午前0時だけではWillMeterDayを終了しない
11. 仮眠はWillMeterDayを終了しない
12. 一日を終える睡眠は、新しいWillMeterDayの開始契機となる
13. 翌日の開始値は前日の終了値と独立して保持する
14. プレミアム権利の有無は記録データそのものを変更しない

### 2.17 代表的な受け入れ条件(Gherkin抜粋)

```gherkin
# 消費記録
Given 現在値が5である
And 「会議」が-3に設定されている
When 利用者が消費として「会議」を記録する
Then 現在値は2になる
And 変化量-3の行動記録が保存される

# 一日を終える睡眠
Given 活動中のWillMeterDayが存在する
When 利用者が一日を終える睡眠を記録する
Then 現在のWillMeterDayは完了状態になる
And アプリはawaitingDayStart状態になる
When 次回起動時に利用者が7を入力する
Then 開始値7の新しいWillMeterDayが作成される

# 基準範囲超過
Given 現在値が9である
When +4の回復を記録する
Then 現在値は13になる
And 数値13が表示される
And メーターは+10超の状態を表示する

# 無料利用者の週次分析
Given プレミアム権利がない
When 週次分析を開く
Then 分析結果は表示されない
And プレミアム説明への導線が表示される
```

### 2.18 WillMeterが扱わないもの

医学的診断、精神状態/ADHD/うつ/睡眠障害の判定、生産性の客観測定、性格診断、行動の善悪判定、タスク/カレンダー管理、ポモドーロタイマー、スマートフォン利用制限、強制的な習慣化、他者との比較・ランキング・連続記録報酬。

## 4. アーキテクチャ方針(Simple First版・このリポジトリでの読み替え)

会話内の技術設計草稿は「Presentation → Application/UseCase → Domain ← Infrastructure」という層構成・Repository protocol一覧・UseCase一覧を提示しているが、このリポジトリは**ADR 0002でSimple Firstを既定方針として採用済み**であり、CLAUDE.mdもこれに従う。したがって本文書では、技術設計草稿の内容を次のように読み替える。

### 4.1 強制しないこと(CLAUDE.mdより継承)

- Domain / Application / Infrastructure / Presentation の形式的な4層分割を新機能追加のたびに厳格適用しない。
- Entity / Repository / Service / UseCase など、DDDパターンの一律導入をしない。
- 実装が1つしかなく、テストや差し替えにも不要なprotocolは作らない。
- 将来の拡張だけを理由にした抽象化・汎用ラッパー・ドメインイベントは追加しない。

### 4.2 守ること

- ドメインロジック(WillPower計算、日境界判定、集計ロジック等)をSwiftUI Viewへ書かず、UI・永続化などのI/Oから分離する。
- ウィルパワー・WillMeterDay・ActivityLogなどのドメイン用語を型名・関数名へ反映する。
- 範囲や不変条件を持つ値(現在値、変化量、好調時の目安等)には、効果が明確な場合にValue Objectを使う。
- 変更理由が異なるコード(例: 日境界判定と表示フォーマット)を小さな責務単位で分ける。
- Repository・protocolは「テストで差し替えが必要」「実装が複数ある」など効果が明確になった時点で導入する(Rule of Three)。

### 4.3 参考: 技術設計草稿のディレクトリ構成案

会話内で提示された構成は次のとおりだが、**強制ルールではなく責務分離の参考例**として扱う。既存の4層ディレクトリ(`WillMeter/Domain` / `Application` / `Infrastructure` / `Presentation`)は既に存在するため直ちに撤去せず、新機能追加時に「この分割が本当に必要か」を都度判断する。

```
WillMeter/
├── App/                  エントリポイント・DI組み立て
├── Domain/
│   ├── Entities/         WillMeterDay, ActivityLog 等
│   ├── ValueObjects/     WillPowerValue, PersonalBaseline 等
│   ├── Services/         日境界判定・集計ロジック(効果が明確な場合のみ)
│   └── Repositories/     protocol(差し替え・テスト用途が明確な場合のみ)
├── Application/          UseCase相当(1関数・1クロージャで足りるなら簡略化)
├── Infrastructure/
│   ├── Persistence/      UserDefaults / JSON
│   └── StoreKit/
└── Presentation/
    ├── Meter/ History/ Settings/ Credits/ ...
```

### 4.4 データ保存方式

- **UserDefaults**: 軽量なUI設定(初回起動済みフラグ、テーマ、モノクロモード、選択中タブ、言語設定)。
- **JSONファイル**: WillMeterDay、ActivityLog、Category、SubCategory、PersonalBaseline、現在状態、Tips閲覧状態。保存時は一時ファイルへ書き込み後に置換し、破損に備え直前世代のバックアップを1世代保持する。
- **StoreKit 2**: プレミアム権利(非消耗型、商品1点)。
- **SwiftDataは採用しない**(データ量が小さくJSONの方が構造を把握しやすい、エクスポートとの親和性が高い、Repositoryを介せば将来移行できるため)。ただしJSONファイルをViewやViewModelから直接操作してはならず、Repository相当を経由する。

推奨ファイル分割(参考、必要になった時点で分割):

```
Application Support/WillMeter/
├── state.json
├── days.json
├── activity_logs.json
├── categories.json
└── metadata.json
```

### 5.1 特筆すべき非互換ポイント

- **WillPowerモデルの全面置き換えが前提**。既存の`WillPower`クラス(0〜100、`maxValue`固定、`consume`が「不足時false」を返す仕様)は新仕様の数値モデル(上限なし・主観値・好調時の目安)と概念からして異なる。段階的移行ではなく作り直しに近い判断が必要になる可能性が高い。
- **既存UserDefaultsデータの扱いが未決定**。新規ユーザーは影響ないが、TestFlight配布済みの既存データ(`willPower.currentValue`等)をどう扱うか(移行/リセット/破棄)は実装着手前に決める必要がある(7章)。
- ソフトウェアのアーキテクチャ大枠(Simple First / ADR 0002)は新仕様と矛盾しないため維持する。
